    üí¨ Mediator Design Pattern¬∂

    üìò Definition¬∂

    The Mediator Design Pattern is a behavioral pattern that defines an object (the mediator) to encapsulate how a set of objects interact.
    It promotes loose coupling by preventing objects from referring to each other directly ‚Äî instead, they communicate through the mediator.

    üéØ Intent¬∂

    To centralize communication between multiple objects so that they don‚Äôt need to explicitly reference each other.

    üß© Real-Life Analogy¬∂

    Consider a chat room where multiple users can send messages.
    Instead of each user directly sending messages to all others, they send it to a ChatRoom (Mediator), which then handles message delivery.
    This makes communication structured and controlled.

    ‚öôÔ∏è Key Participants¬∂

    Participant Role
    Mediator (Interface) Defines communication methods between objects.
    ConcreteMediator Implements communication logic between objects.
    Colleague (Component) Represents individual objects that communicate via mediator.
    ConcreteColleague Concrete implementation of a colleague class.

    üìä UML Diagram¬∂

    classDiagram
    class Mediator {
        <<interface>>
        + sendMessage(msg: string, sender: Colleague)
    }

    class ConcreteMediator {
        - users : list~Colleague*~
        + addUser(user: Colleague)
        + sendMessage(msg: string, sender: Colleague)
    }

    class Colleague {
        <<abstract>>
        + send(msg: string)
        + receive(msg: string)
    }

    class User {
        - name : string
        - mediator : Mediator*
        + send(msg: string)
        + receive(msg: string)
    }

    Mediator <|.. ConcreteMediator
    Colleague <|.. User
    ConcreteMediator --> Colleague
    User --> Mediator

    üíª C++ Implementation¬∂

    mediator-design.cpp
     1#include <iostream>
 2#include <vector>
 3#include <string>
 4#include <memory>
 5using namespace std;
 6
 7// Forward declaration
 8class User;
 9
10// Mediator Interface
11class ChatMediator {
12public:
13    virtual void sendMessage(const string& msg, User* sender) = 0;
14    virtual void addUser(shared_ptr<User> user) = 0;
15    virtual ~ChatMediator() = default;
16};
17
18// Colleague (User) Interface
19class User {
20protected:
21    string name;
22    ChatMediator* mediator;
23public:
24    User(string n, ChatMediator* m) : name(move(n)), mediator(m) {}
25    virtual void send(const string& msg) = 0;
26    virtual void receive(const string& msg) = 0;
27    string getName() const { return name; }
28    virtual ~User() = default;
29};
30
31// Concrete Mediator
32class ConcreteChatMediator : public ChatMediator {
33    vector<shared_ptr<User>> users;
34public:
35    void addUser(shared_ptr<User> user) override {
36        users.push_back(user);
37    }
38
39    void sendMessage(const string& msg, User* sender) override {
40        for (auto& user : users) {
41            if (user->getName() != sender->getName())
42                user->receive(sender->getName() + ": " + msg);
43        }
44    }
45};
46
47// Concrete User (Colleague)
48class ChatUser : public User {
49public:
50    ChatUser(string n, ChatMediator* m) : User(move(n), m) {}
51    void send(const string& msg) override {
52        cout << name << " sends: " << msg << endl;
53        mediator->sendMessage(msg, this);
54    }
55
56    void receive(const string& msg) override {
57        cout << name << " receives -> " << msg << endl;
58    }
59};
60
61// Client Code
62int main() {
63    ConcreteChatMediator chatRoom;
64
65    auto user1 = make_shared<ChatUser>("Alice", &chatRoom);
66    auto user2 = make_shared<ChatUser>("Bob", &chatRoom);
67    auto user3 = make_shared<ChatUser>("Charlie", &chatRoom);
68
69    chatRoom.addUser(user1);
70    chatRoom.addUser(user2);
71    chatRoom.addUser(user3);
72
73    user1->send("Hello, everyone!");
74    user2->send("Hi Alice!");
75    return 0;
76}

    üß† Output¬∂

    1Alice sends: Hello, everyone!
2Bob receives -> Alice: Hello, everyone!
3Charlie receives -> Alice: Hello, everyone!
4Bob sends: Hi Alice!
5Alice receives -> Bob: Hi Alice!
6Charlie receives -> Bob: Hi Alice!

    ‚úÖ Advantages¬∂

    • Loose coupling ‚Äî objects don‚Äôt need to know about each other directly.
    • Centralized control ‚Äî all communication is managed in one place.
    • Easy to add new participants without modifying existing classes.

    ‚ö†Ô∏è Disadvantages¬∂

    • The mediator can become complex as the number of interactions grows.
    • Centralized logic may lead to a God Object if not designed carefully.

    üí° Use Cases¬∂

    • Chat applications or messaging systems.
    • Air traffic control systems (planes communicate via control tower).
    • UI event coordination (buttons, forms, modals).

    üß± System Design View¬∂

    In large systems:

    • The Mediator can be implemented as a Message Bus or Event Dispatcher.
    • Helps integrate microservices or components without tight dependencies.
