    üè™ Zepto | 10-Minute Quick Commerce | Inventory Management | System Design¬∂

    üöÄ Overview¬∂

    A 10-minute delivery system like Zepto requires a tightly coupled ecosystem of microservices, optimized inventory distribution, and real-time order orchestration. This design focuses on scalability, availability, and low-latency response through efficient service communication and caching.

    üß† Core Idea¬∂

    Deliver essential goods within 10 minutes by connecting:

    • Local dark stores (warehouses)
    • Nearby delivery partners
    • Real-time inventory
    • Smart routing & order allocation

    ‚öôÔ∏è Key Components¬∂

    1. User App - Allows users to browse inventory, add to cart, and place orders. - Tracks order status and estimated delivery time. = Communicates with backend via API Gateway.

    2. API Gateway - Central entry point for all client requests. - Routes requests to appropriate microservices (Order, Inventory, Payment, etc.). - Handles authentication, rate limiting, and request aggregation.

    3. Order Service - Manages order lifecycle: creation, validation, and coordination. - Validates inventory and triggers payment. - Updates status to Notification Service and Delivery Service. - Stores order metadata for analytics.

    4. Inventory Service - Tracks product availability across warehouses. - Provides instant availability checks using Redis cache. - Updates inventory in real time after order allocation. - Syncs periodically with Warehouse Service.

    5. Warehouse Service - Represents dark stores (micro-warehouses). - Handles product restocking, stock transfers, and expiry checks. - Collaborates with Location Mapper to identify nearest fulfillment hub.

    6. Location Mapper - Maps customer delivery address to nearest active warehouse. - Uses geo-coordinates or pincode mapping for minimal delivery latency. - Balances load among nearby warehouses.

    7. Payment Service - Processes payments securely via third-party gateways. - Handles transaction retries, refunds, and payment status updates. - Notifies Order Service once payment succeeds or fails.

    8. Delivery Logistics Service - Assigns delivery partner (rider) from nearest hub. - Optimizes route planning and expected delivery time. - Tracks delivery progress and updates customer in real-time.

    9. Notification Service - Sends updates via SMS, Email, or Push Notifications. - Triggered by order, payment, or delivery events. - Uses asynchronous messaging queues for reliability.

    10. Analytics & Dashboard - Aggregates data from all services for business insights. - Tracks order metrics, delivery time, and stock movement. - Provides Admins a real-time monitoring dashboard.

 28std::string now() {

 53class PaymentService {

 55    bool processPayment(int orderId, double amount) {
 56        cout << "[" << now() << "] [PaymentService] Processing payment for Order#" << orderId
 57             << " Amount: " << amount << "\n";
 58        // simulate network delay
 59        std::this_thread::sleep_for(std::chrono::milliseconds(150 + rand()%200));

 67// Warehouse (dark store) with stock
 68struct Warehouse {
 69    string id;
 70    string location; // friendly name
 71    unordered_map<string,int> stock; // sku -> qty
 72    Warehouse(const string& i, const string& loc): id(i), location(loc) {}
 73};
 74
 75class InventoryService {
 76public:
 77    InventoryService() {}
 78
 79    // add warehouse
 80    void addWarehouse(shared_ptr<Warehouse> wh) {
 81        std::lock_guard<std::mutex> lk(mu_);
 82        warehouses_.push_back(wh);
 83        // fill cache for quick lookup (sum of stocks)
 84        rebuildCache();
 85    }
 86
 87    // check availability: returns vector of warehouses that can fulfill quantity
 88    vector<shared_ptr<Warehouse>> findWarehouses(const string& sku, int qty) {
 89        std::lock_guard<std::mutex> lk(mu_);
 90        vector<shared_ptr<Warehouse>> res;
 91        for (auto &wh : warehouses_) {
 92            auto it = wh->stock.find(sku);
 93            if (it != wh->stock.end() && it->second >= qty)
 94                res.push_back(wh);
 95        }
 96        return res;
 97    }
 98
 99    // allocate sku from specific warehouse (reduce stock)
100    bool allocateFromWarehouse(shared_ptr<Warehouse> wh, const string& sku, int qty) {
101        std::lock_guard<std::mutex> lk(mu_);
102        auto it = wh->stock.find(sku);
103        if (it == wh->stock.end() || it->second < qty) return false;
104        it->second -= qty;
105        // update cache
106        cache_[sku] -= qty;
107        if (cache_[sku] < 0) cache_[sku] = 0;
108        cout << "[" << now() << "] [InventoryService] Allocated " << qty << " of " << sku
109             << " from Warehouse[" << wh->id << "]\n";
110        return true;
111    }
112
113    // quick availability check via 'cache' (simulated Redis)
114    int availableCache(const string& sku) {
115        std::lock_guard<std::mutex> lk(mu_);
116        return cache_.count(sku)? cache_[sku] : 0;
117    }
118
119    void rebuildCache() {
120        cache_.clear();
121        for (auto &wh : warehouses_) {
122            for (auto &p : wh->stock) cache_[p.first] += p.second;
123        }
124    }
125
126    vector<shared_ptr<Warehouse>> getAllWarehouses() {
127        std::lock_guard<std::mutex> lk(mu_);
128        return warehouses_;
129    }
130
131private:
132    vector<shared_ptr<Warehouse>> warehouses_;
133    unordered_map<string,int> cache_;
134    std::mutex mu_;
135};

137// Delivery service: assign rider and simulate delivery
138class DeliveryService {
139public:
140    DeliveryService(NotificationService& notifier): notifier_(notifier) {}
141
142    void assignAndDeliver(int orderId, const string& customer, shared_ptr<Warehouse> wh) {
143        // simulate assignment
144        cout << "[" << now() << "] [DeliveryService] Assigning rider for Order#" << orderId
145             << " from Warehouse[" << wh->id << "]\n";
146        notifier_.notifyCustomer(orderId, "Rider assigned. Picking up from " + wh->location);
147        // simulate rider pickup & transit
148        std::this_thread::sleep_for(std::chrono::milliseconds(200 + rand()%500));
149        notifier_.notifyCustomer(orderId, "Order out for delivery");
150        // simulate delivery time (we'll keep it short for demo)
151        std::this_thread::sleep_for(std::chrono::milliseconds(400 + rand()%800));
152        notifier_.notifyCustomer(orderId, "Delivered ‚Äî Enjoy!");
153        cout << "[" << now() << "] [DeliveryService] Order#" << orderId << " delivered to " << customer << "\n";
154    }
155
156private:
157    NotificationService& notifier_;
158};

160// Order Service controlling flow and orchestrating other services
161class OrderService {
162public:
163    OrderService(InventoryService& inv, PaymentService& pay, DeliveryService& dl, NotificationService& ns)
164        : inv_(inv), pay_(pay), delivery_(dl), notifier_(ns) {}
165
166    struct OrderRequest {
167        string customerName;
168        string sku;
169        int qty;
170        double amount;
171    };
172
173    void placeOrder(const OrderRequest& req) {
174        int orderId = GLOBAL_ORDER_ID.fetch_add(1);
175        cout << "[" << now() << "] [OrderService] Received order request. Order#" << orderId
176             << " Customer: " << req.customerName << " SKU: " << req.sku << " Qty: " << req.qty << "\n";
177
178        // Notify received
179        notifier_.notifyCustomer(orderId, "Order received. Checking availability...");
180
181        // Step 1 - Quick cache check (Redis-like)
182        int cachedAvail = inv_.availableCache(req.sku);
183        cout << "[" << now() << "] [OrderService] Cache availability for " << req.sku << " = " << cachedAvail << "\n";
184
185        if (cachedAvail < req.qty) {
186            // fallback to checking warehouses specifically
187            auto whs = inv_.findWarehouses(req.sku, req.qty);
188            if (whs.empty()) {
189                notifier_.notifyCustomer(orderId, "Item out of stock. Sorry!");
190                cout << "[" << now() << "] [OrderService] Order#" << orderId << " failed: OOS\n";
191                return;
192            }
193        }
194
195        // Step 2 - Process payment (synchronously)
196        notifier_.notifyCustomer(orderId, "Processing payment...");
197        bool paid = pay_.processPayment(orderId, req.amount);
198        if (!paid) {
199            notifier_.notifyCustomer(orderId, "Payment failed. Please try again.");
200            cout << "[" << now() << "] [OrderService] Order#" << orderId << " failed: Payment\n";
201            return;
202        }
203
204        // Step 3 - Allocate from nearest warehouse (simplified: choose first with stock)
205        auto whs = inv_.findWarehouses(req.sku, req.qty);
206        if (whs.empty()) {
207            // rare race: paid but stock gone
208            notifier_.notifyCustomer(orderId, "Paid, but item became out-of-stock. Initiating refund.");
209            // refund logic omitted
210            cout << "[" << now() << "] [OrderService] Order#" << orderId << " failed after payment: Race OOS\n";
211            return;
212        }
213
214        // For demo choose the first warehouse (in real system we'd use LocationMapper + distance)
215        auto chosenWH = whs.front();
216        bool allocated = inv_.allocateFromWarehouse(chosenWH, req.sku, req.qty);
217        if (!allocated) {
218            notifier_.notifyCustomer(orderId, "Allocation failed. We'll refund you.");
219            cout << "[" << now() << "] [OrderService] Order#" << orderId << " allocation failure\n";
220            return;
221        }
222
223        notifier_.notifyCustomer(orderId, "Order confirmed. Preparing for dispatch.");
224
225        // Step 4 - push to delivery queue (simulate asynchronous delivery)
226        std::thread deliveryThread([this, orderId, customer=req.customerName, chosenWH]() {
227            // small delay to mimic queue processing
228            std::this_thread::sleep_for(std::chrono::milliseconds(100 + rand()%200));
229            this->delivery_.assignAndDeliver(orderId, customer, chosenWH);
230        });
231        deliveryThread.detach();
232
233        cout << "[" << now() << "] [OrderService] Order#" << orderId << " processed successfully\n";
234    }
235
236private:
237    InventoryService& inv_;
238    PaymentService& pay_;
239    DeliveryService& delivery_;
240    NotificationService& notifier_;
241};

244int main() {
245    srand((unsigned)time(nullptr));
246    cout << "Zepto-like Quick Commerce Simulation Starting...\n";

248    // Services

250    InventoryService inventory;
251    PaymentService payment;

257    auto wh2 = make_shared<Warehouse>("WH-002", "Central Park Darkstore");
258
259    // Populate stock: SKUs: MILK1, BUNS1, CHOC1
260    wh1->stock["MILK1"] = 10;
261    wh1->stock["BUNS1"] = 20;
262    wh1->stock["CHOC1"] = 5;
263
264    wh2->stock["MILK1"] = 5;
265    wh2->stock["BUNS1"] = 0;
266    wh2->stock["CHOC1"] = 12;
267
268    inventory.addWarehouse(wh1);
269    inventory.addWarehouse(wh2);
270
271    // Simulate incoming orders through API Gateway (multiple threads)
272    vector<OrderService::OrderRequest> requests = {
273        {"Alice", "MILK1", 2, 80.0},
274        {"Bob",   "CHOC1", 3, 150.0},
275        {"Carol", "BUNS1", 5, 120.0},
276        {"Dave",  "CHOC1", 7, 350.0}, // might fail due to stock
277        {"Eve",   "MILK1", 8, 320.0}
278    };
279
280    vector<thread> clients;
281    for (size_t i = 0; i < requests.size(); ++i) {
282        clients.emplace_back([&, i]() {
283            // random small delay between orders
284            std::this_thread::sleep_for(std::chrono::milliseconds(50 * (int)i + (rand()%100)));
285            orderService.placeOrder(requests[i]);
286        });
287    }
288
289    // Wait for clients to finish issuing orders
290    for (auto &t : clients) if (t.joinable()) t.join();
291
292    // Wait some time for delivery threads to finish (demo only)
293    std::this_thread::sleep_for(3s);
294
295    cout << "Simulation complete.\n";
296    return 0;
297}

    üíæ Data Management¬∂

    Service Storage Type Description
    Inventory NoSQL + Redis Cache Product availability, stock count
    Order SQL Order metadata, timestamps, status
    Payment SQL Transaction logs, payment status
    Warehouse NoSQL Warehouse location & stock details
    Analytics Data Warehouse Aggregated performance metrics

    üîÑ System Flow¬∂

    1. User places order via app ‚Üí API Gateway routes to Order Service.
    2. Order Service validates product availability from Inventory Service.
    3. Inventory Service checks cache or warehouse data.
    4. If available ‚Üí Payment Service processes payment.
    5. On success ‚Üí Warehouse Service allocates items.
    6. Delivery Logistics assigns nearest rider and dispatches order.
    7. Notification Service updates customer throughout lifecycle.
    8. Analytics Service logs data for performance monitoring.

    ‚ö° Performance & Scalability¬∂

    Concern Approach
    Low latency Redis caching for instant inventory lookup
    Scalability Microservice-based modular design
    Reliability Message queues for async communication
    Availability Multi-zone warehouse and delivery routing
    Data consistency Event-driven updates with idempotent APIs

    üß© Extensibility¬∂

    Feature How to Extend
    New delivery partner Plug-in new service into Delivery API
    Add new notification channel Extend Notification Adapter
    Support new payment mode Add Payment Gateway Adapter
    Expand region Add warehouses and update Location Mapper

    üß± Design Principles Used¬∂

    • Single Responsibility Principle (SRP): Each service has one core job.
    • Command Design Pattern: Encapsulates order actions as commands.
    • Adapter Pattern: Integrates third-party services (SMS, Payment).
    • Observer/Event Pattern: Keeps services loosely coupled via notifications.
    • Cache-Aside Strategy: Load inventory into cache for faster lookups.

    ‚úÖ Summary¬∂

    This architecture ensures:

    • Ultra-fast order-to-dispatch pipeline.
    • Real-time stock accuracy.
    • Scalable and modular microservices.
    • Seamless integration with delivery and payment ecosystems.

    The design achieves Zepto‚Äôs 10-minute delivery promise by combining smart inventory distribution, low-latency service orchestration, and real-time analytics.
